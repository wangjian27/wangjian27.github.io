高性能渲染长列表

高性能渲染长列表

背景

  • 在平常的业务当中，总避免不了出现大量数据需要渲染的情况
  • 在处理这些大量数据渲染的时候，我们一般的做法大多都是一种分页懒加载的做法

分页加载的缺点与不足:

  • 1.用户在向下滑动的过程中请求接口需要等待
  • 2.越往下滑动加载出了很多的dom，造成了性能的损耗
  • 3.当渲染出来的列表数据需要时刻改变数据的时候，由于回流的发生，大量的dom造成大量的性能损耗。移动端的直观表现就是手机的发烫和耗电的大大增加

要解决的两个点：

  • 1.让用户在不用等待下拉加载的过程
  • 2.减少性能损耗

一次性渲染

  • 优点：解决了用户下拉加载等待的问题
  • 缺点：首次加载时间过长

分批渲染(一)

  • js是单线程运行，js在触发渲染线程的时候会阻塞，所以通过一种分批渲染的方式，让js不阻塞线程
  • 优点：以一种递归渲染的方式插入dom，大大减少了用户首次加载时间(就是插入20条的时间)
  • 缺点：快速滚动会有闪屏卡屏的情况

分批渲染(二)

  • 由于setTimeout的执行时间是不确定的，在渲染数据的时候，不能够与屏幕的刷新时机相匹配
  • 因为requestAnimationFrame是在屏幕每一帧之后运行一次，所以将setTimeout改为requestAnimationFrame。
  • 优点：解决了闪屏情况
  • 缺点：在循环之中添加dom会造成大量回流

分批渲染(三)

  • 使用documentFragment
    • 1.documentFragment不是真实dom树一部分，他不会触发dom树的重新渲染。
    • 2.可以通过document.createDocumentFragment创造出的存在内存中的dom
  • 优点：1.解决了用户下拉加载的问题; 2.解决了用户首次加载时间过长的问题; 3.且在渲染的过程中优化了性能
  • 缺点：渲染了大量的dom，依旧会造成性能上的问题

虚拟列表–基本介绍

  • 滚动过去和未滚动的列表数据其实都是一些无效的数据，真正想要的是可视区域这一块dom。所以要做到只渲染可视区域这一块儿dom，这样dom就会很少，大大的减少了性能的消耗。

虚拟列表–具体实现原理

  • 虚拟列表的实现，实际上就是在首屏加载的时候，只加载可视区域内需要的列表项，当滚动发生时，动态通过计算获得可视区域内的列表项，将这些列表项的数据去不断替换原来可视区域的列表项。
  • 1.根据行高及总条数计算出一个总高度，并用一个dom撑开这么高的高度，让滚动条能够滚动
  • 2.计算当前可视区域起始数据索引(start)
  • 3.根据列表高度计算可视区域列表行数(visibleCount)
  • 4.计算当前可视区域结束数据索引(end)
  • 5.将开始结束这一段之间的数据取出渲染到可视区域上
  • 6.为列表添加滚动事件，根据向上滚动的距离，将可视区域通过transform的方式，向下偏移，让可视区域一直展示在屏幕上

虚拟列表的业务使用

    • 1.长列表(无法直接展示)
    • 2.股票及行情数据实时刷新(无法用懒加载)

总结:

  • 优点:
    • 1.虚拟列表确实能解决懒加载下拉等待的过程，优化了用户的体验；
    • 2.相对于直接加载和分批加载来说确实提升了性能；
  • 局限性:
    • 1.接口返回的全量数据对接口的带宽有一定的要求（最好能有客户端协议支持）

前端性能优化

为什么要性能优化？

  • 前端性能优化提升用户使用友好程度和提高用户留存率的一个重要属性，也是一个前端工程师进阶的一个重要指标，所以总结一下性能优化的技巧十分有必要。

性能优化方面细分

一、网络请求方面

  • 1.完整的http请求需要经历DNS的查询、tcp的握手等过程，在这个过程中，不仅有DNS服务器解析数据的时间，还有http请求、服务端处理请求并相应、最终返回给客户端的过程。接下来看具体的例子：
    • 结论：以上是一个接口请求整体耗费的时间，如果减少请求的次数，就会相应的减少DNS查询时间和tcp握手时间。
  • 2.使用http2
  • 3.SSR
  • 4.静态资源使用CDN
  • 5.善于使用缓存

二、代码层面

  • 1.尽量把css文件放在头部，js文件放在底部，因为浏览器是单线程工作
    • CSS 执行会阻塞渲染，阻止 JS 执行
    • JS 加载和执行会阻塞 HTML 解析，阻止 CSSOM 构建

如果css和js都放在head里，就会导致加载和解析很久，用户就会看到一段空白页面。因此把js放在底部，不阻塞dom的解析。把css放在顶部，让渲染出来的dom都有样式，防止用户看到没有css样式的dom。

  • 2.减少在循环中使用全局变量

用node搭建属于自己的脚手架

自定义脚手架工具解决问题

  • 1.可以自定义需要导入的文件；
  • 2.自定义框架选择；
  • 3.减少自己的一些重复性的工作。

    脚手架的整体流程

  • 1.根据用户输入来生成不同的框架配置文件；
  • 2.下载指定后的模板；
  • 3.根据用户信息的不同，生成不同package.json；
  • 4.生成新的工程文件。

    脚手架需要用到的依赖

  • 1.commander：参数解析工具。用它来读取用户输入的数据，并且根据用户输入的数据来进行一定的操作。
  • 2.inquirer：交互式的命令行工具。用它通过选择、读取、判断的方式读取用户的数据。
  • 3.download-git-repo：下载远程模板的工具。用它来下载在git上配置好的模板。
  • 4.ora：用户显示加载中的效果。显示一个loading的效果，提升用户交互友好度。
  • 5.chalk：将console.log打印的内容颜色进行修改。

    开发步骤

    一、初始化

  • 1.创建项目文件夹；
  • 2.npm init –y初始化项目;
  • 3.创建入口文件;
  • 4.并在package.json中追加’bin’的链接的入口文件地址；
  • 5.用npm link链接到全局。
    **package.json**

二、使用commander处理参数

  • 1.npm install commander；
  • 2.const program=require(‘commander’)

三、在git上创建模板仓库

  • 1.可以在git上建立组织，然后只下载这个组织内的模板；
  • 2.在cli create的时候，遍历这个组织的仓库以及tags，用inquirer
    进行命令行问答，达到一种动态配置的效果。

四、使用download-git-repo从git上下载模板

  • 1.npm install download-git-repo ；
  • 2.const downloadGitRepo=require(‘download-git-repo’)
  • 3.tips：download-git-repo不是promise对象，可以用util来对异步promise化，然后就可以愉快的使用async、await了

五、收集用户数据，生成package.json

六、发布到npm库

  • 1.完成之后，npm login登录npm；
  • 2.npm publish发布到npm库。
  • 注意:
    • 1.发布前查是否已存在同名包;
    • 2.有淘宝镜像的切换回官方。
    • 3.每回发布需要更改version
  • tips：
    • 可以下载nrm来管理自己的npm源
